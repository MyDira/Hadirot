# SMS Systems Audit — July 8, 2026

Full top-to-bottom audit of every SMS system: code (8 edge functions + shared token lib + client), schema/migrations, cron scheduling, and **live production data** (direct read-only queries against prod `pxlxdlrjmrkxyygdhvku` via `SUPABASE_DB_URL`, including `cron.job_run_details` history, `sms_messages` traffic, and `listing_renewal_conversations` outcomes).

## Verdict

The core loops fixed in the July 2 launch work (inbound webhook alive, state constraint fixed, callback/report conversations now persist, boost upsell firing). But **two entire SMS systems have never sent a single message** because their cron jobs were never wired correctly, and the renewal-reminder system silently skips ~2 of every 7 listings and never asks multi-listing owners about most of their listings.

Engagement reality check (90 days of renewal conversations): **310 timed out, 16 extended, 3 deactivated** — ~6% of conversations reach an outcome.

## P0 — Broken in production right now

### 1. `send-paid-listing-reminders` has NEVER run — 0 payment-reminder SMS ever sent
- Prod evidence: **28 of 28 cron runs failed** in the last 30 days, every one with `ERROR: unrecognized configuration parameter "app.supabase_url"`. `SELECT max(created_at) FROM sms_messages WHERE message_source='paid_listing_reminder'` → **NULL** (never, ever).
- Cause: migration `20260527150600_schedule_paid_listing_reminders.sql` builds the URL from `current_setting('app.supabase_url')`, and **no `app.*` settings exist in prod** (`SELECT name FROM pg_settings WHERE name LIKE 'app.%'` → empty). The migration's own comment says these "must be configured once on the Supabase project" — that step never happened.
- Impact: monetization is **ON** (`monetization_enabled=true`), yet trial-ending, paid-balance-ending, and post-deactivation reminders — the whole "pay from your phone" revenue loop, including the signed pay-link infrastructure (`sms-link-token.ts`, `pay-listing-link`) — is dead. `pay-listing-link` has effectively never received real traffic.
- Fix: reschedule the cron with a hardcoded URL + service-role key header exactly like the working `send-renewal-reminders` job (jobid 13), or set the two GUCs. One-line SQL either way.
- Bonus bug once revived: the schedule is `0 10 * * *` = 10:00 **UTC** = **6 AM ET**, despite the migration comment saying "10 AM Eastern". Reminder texts would arrive at 6am. Change to `0 14 * * *`.

### 2. `send-weekly-performance-reports` has NEVER run — 0 weekly report SMS ever sent
- Prod evidence: cron job 17's command still contains the literal placeholder **`https://[PROJECT-REF].supabase.co/...`** from migration `20260121191812` (Jan 21, 2026). All 4 runs in the last 30 days failed (libcurl protocol error on the invalid URL). `max(created_at) WHERE message_source='weekly_report'` → **NULL**.
- Impact: the "This week your listings got X impressions / Y leads" owner-engagement SMS has silently never existed, since January.
- Fix: `cron.alter_job`/reschedule with the real URL + service key.

## P1 — Systemic design flaws with live impact

### 3. Shabbat skip permanently drops ~2/7 of renewal reminders
`send-renewal-reminders` targets listings expiring **exactly 5 days from today** and returns early on Friday/Saturday (NY). There is no catch-up: a listing whose expiry lands 5 days after a Friday or Saturday (i.e., expires Wed/Thu) **never gets any renewal SMS** and expires unasked. Fix: on Sunday, widen the window to cover the skipped days (expiring in 3–5 days), or shift the skipped cohorts to Sunday.

### 4. Multi-listing owners never get asked about most of their listings
The batch flow only ever sends the question for listing #1; listings 2..N sit in `pending` and advance one at a time **only if the owner replies**. With a ~6% reply rate, prod shows **130 of 310 renewal timeouts in 90 days were position-≥2 listings that were never messaged at all** (42%). Because the sender's window is "exactly 5 days out", these listings never re-qualify — they silently expire. Related edge (code-proven): if the first-in-batch Twilio send fails or is dedup-skipped, the entire batch is stranded in `pending` until timeout. Fix options: send batch question listing-by-listing on subsequent days, or timeout-advance the batch, or ask "reply YES to keep ALL N listings".

### 5. Renewal SMS conflicts with monetization lifecycle
- `send-renewal-reminders` doesn't filter by `payment_kind` — trial/paid listings get "Reply YES to extend" (2 `individual_trial` conversations in last 30 days). A YES sets `expires_at += 14d`, but `auto_inactivate_old_listings()` deactivates trials at `trial_started_at+14d` and paid listings at `paid_until` **regardless** — the owner is told "extended for another 2 weeks" and the listing dies anyway at midnight. And since the paid-reminder system (#1) is dead, the misleading free-extension SMS is the *only* reminder monetized owners receive.
- Opposite direction for free listings: the YES handler sets `last_published_at=now`, and deactivation uses `GREATEST(expires_at, last_published_at + rental_active_days)` with `rental_active_days=40` → the SMS says "will expire on ⟨+2 weeks⟩" but the listing actually lives **40 days**. The quoted expiry date is wrong in both directions.
- Fix: exclude `individual_trial`/`individual_paid`/`pending_payment` from the availability-renewal sender (route them to the pay-link reminders once #1 is fixed), and make the confirmation copy reflect actual lifecycle.

## P2 — Abuse & robustness gaps

### 6. `send-listing-contact-sms` is unauthenticated-in-practice with zero rate limiting
The client calls it with the **anon key** as the Bearer token, which passes `verify_jwt`. Anyone can loop the endpoint and (a) SMS-bomb any listing owner, (b) burn Twilio spend, (c) pollute `listing_contact_submissions` (`ip_address` is client-supplied, so even the audit trail is spoofable). Fix: per-phone/per-listing cooldown check against `sms_messages`/`listing_contact_submissions` (e.g., max 3 per listing per hour), server-derived IP.

### 7. No Twilio delivery-status callbacks — carrier filtering is invisible
Every outbound row stays `status='sent'` forever (prod: only `sent`/`received` exist). If A2P carrier filtering blocks messages, nothing ever shows it. Notably **54 of 79 phones messaged in the last 30 days have never replied to anything** — impossible to distinguish disinterest from non-delivery. Fix: pass `StatusCallback` on Twilio sends + a small status-webhook edge function updating `sms_messages.status`; alert admin on `undelivered` spikes. (Twilio console A2P registration check is still on the user's pending list from July 2.)

### 8. Failure-logging inconsistencies (latent)
- `handle-renewal-sms-webhook`'s `sendSMS()` logs the message with `status:'sent'` even when Twilio returned an error (SID null). No occurrences yet in prod, but failures would be recorded as successes.
- `send-renewal-reminders` conversation-insert failures only `console.error` — no admin email alert, unlike the contact/report functions which got `alertAdminSmsFailure` in the July 2 fix. The constraint bug that hid for months is exactly this class.
- Renewal-reminder `sms_messages` rows are never linked to their conversation (`conversation_id` stays NULL; the report function does link).

## P3 — Minor

9. **"Deactivate in 24 hours" is up to ~24h late**: report-rented SMS promises deactivation in 24h, but `cleanup-expired-renewals` runs once daily at midnight UTC.
10. **Weekly reports group by raw `contact_phone`** (not E.164-normalized) — same owner with differently-formatted phones across listings gets split/duplicate reports. (Moot until #2 is fixed.) Stale comment there also claims commercial views don't flow into analytics_events — they do since the July 2 parity fix.
11. **Server rejects 11-digit phone input** in contact SMS (e.g., "1-718-555-1234"): server requires exactly 10 digits after cleaning. Client enforces 10 digits too, so consistent — but any direct/API caller with a country code gets a 400.
12. **`notifyAdmin` in the webhook ignores `notify_on_errors`/`notify_on_unrecognized`** config flags — it selects them but never checks them.
13. **Sale listings get the same "Reply YES to extend 14 days" flow** (39 conversations in 90d) with `sale_active_days=65` — same copy/lifecycle mismatch as #5, deliberate-ish but worth a product decision.
14. Dead code: `void listingTypeWord;` in the sender; duplicated `formatSpaceType` maps across 4 functions (drift risk — the report function's copy already diverges: no `flex`/`retail` keys).
