# Track 5 — Architecture, Duplication & Spaghetti Map — Findings

## Summary

Verdict: this is duplication-driven spaghetti, not tangled-dependency spaghetti — and the two compound very differently. There is almost no circular-dependency mess; the router is clean, the service layer exists and is mostly (not fully) respected, and the two known "past incident" bug classes (filter persistence, price filter) were both fixed in genuinely shared code (`useBrowseFilters.ts`, `ListingFiltersHorizontal.tsx`), so those fixes protect both twins. The actual disease is **parallel reimplementation**: nearly every feature exists twice — once for residential, once for commercial/sale — as independent files with independent bugs, rather than as one parameterized implementation. This is visible in the posting flows (2 live commercial-posting UIs, ~4,661 lines combined), the Browse pages (91.5% line-identical, drifted in "active filters" detection and in map-view favoriting), and a residential/commercial twin pattern that repeats across detail pages, similar-listings widgets, and map popups. Each twin pair is a place a future fix can be applied to one side and forgotten on the other — which is exactly the class of bug this codebase's own `.md` history (FILTER_PERSISTENCE_BUG_FIX.md, PRICE_FILTER_BUG_FIX.md) shows has already happened. Type safety is eroding fastest exactly where the duplication is worst (posting-flow submit boundaries: 45 and 34 `as any` casts in the two commercial posting files alone). Findings: 2 P1, 9 P2, 4 P3.

## Findings

### [P1] Two independently-maintained commercial posting UIs are both live in prod simultaneously

  // redirects to the wizard while off). Flip to true at commercial launch.

Remove the `navigate('/post-commercial')` link from `src/pages/PostListing.tsx:1425` and the `/post-commercial` route registration in `src/App.tsx`. Do NOT delete `src/pages/postCommercial/commercialTypes.ts` or `typeFieldConfigs.ts` — the wizard imports these directly and must keep them. ### [P2] `src/services/listings.ts` is the highest-fan-in file in the codebase and the riskiest to touch
- **Where:** `src/services/listings.ts` (2,092 lines, one of the 10 god files)
- **What:** A repo-wide reference count shows `listings.ts` is imported by roughly 80 other files — an order of magnitude more than any other god file (`AdminPanel.tsx`: 3, `PostListing.tsx`: 7, `EditListing.tsx`: 5, `ListingsMapEnhanced.tsx`: 3, `Dashboard.tsx`: 7, `ListingDetail.tsx`: 6, `BrowseListings.tsx`: 3, `ListingFiltersHorizontal.tsx`: 2, `BrowseSales.tsx`: 2). ### [P2] Two competing phone-number formatters with genuinely different output, not just duplicated code
- **Where:** `src/utils/formatters.ts:32` `formatPhoneNumber()` vs. `src/utils/phone.ts:1` `formatPhoneForDisplay()`
- **What:** Unlike `formatPrice`, these two are not identical. `formatPhoneForDisplay` (phone.ts) handles both a bare 10-digit number (`(area) pre-line`) and an 11-digit number with a leading country code `1` (`+1 (area) pre-line`). `formatPhoneNumber` (formatters.ts) only handles the 10-digit case; for an 11-digit number (e.g. a number stored with a leading `+1` or `1`) it falls through to `return phone` — i.e. returns the raw, unformatted input.
- **Why it matters / failure scenario:** If any phone number in the `profiles`/`listings` tables is ever stored with a leading `1` (a common outcome of Twilio-formatted numbers or a user pasting a "+1 555..." number), whichever component imports `formatPhoneNumber` instead of `formatPhoneForDisplay` will display the raw unformatted digit string instead of a formatted number — a real, user-visible inconsistency depending purely on which of the two files a given component happened to import.
- **Evidence:**
  ```
  // formatters.ts — falls through unformatted for anything but exactly 10 digits
  if (cleaned.length === 10) { return `(${cleaned.slice(0,3)}) ${cleaned.slice(3,6)}-${cleaned.slice(6)}`; }
  return phone;

  // phone.ts — explicitly handles the 11-digit/leading-1 case
  if (digits.length === 11 && digits.startsWith("1")) { return `+1 (${area}) ${pre}-${line}`; }
  if (digits.length === 10) { return `(${area}) ${pre}-${line}`; }
  ```
- **Fix prompt:** Audit every import of `formatPhoneNumber` from `src/utils/formatters.ts` (`grep -rn "formatPhoneNumber" src`) and switch each call site to `formatPhoneForDisplay` from `src/utils/phone.ts`, then delete `formatPhoneNumber` from `formatters.ts` once it has zero remaining importers. Verify by finding a phone number stored with a leading `1` in the DB (or temporarily entering one) and confirming it now displays formatted everywhere it's shown.

### [P2] `src/types/database.ts` (generated) is correctly wired into the Supabase client, but joins and filter objects routinely escape it via `any`
- **Where:** `src/config/supabase.ts:2,15` (`createClient<Database>(...)`); `src/services/listings.ts` and `src/services/commercialListings.ts` (highest `any` counts among services: 7/12 and 9/10 respectively)
- **What:** This is a nuanced, partially-good finding: the generated `Database` type from `src/types/database.ts` (4,558 lines) IS threaded into `createClient<Database>`, so every raw `.from('table').select()` call site does get generated row types automatically — this is not the "generated types exist but nobody uses them" failure mode the master plan worried about. Where type safety actually erodes is (a) Supabase nested-join result shapes (`.select('*, owner:profiles(agency)')`-style queries), which the generated types represent as complex nested optional types that get cast to `any` at the point of use rather than typed precisely, and (b) domain-specific `FilterState`/form-data types that don't declare every property actually read/written, forcing `(filters as any)?.agency_name`-style escapes.

**Is this "spaghetti that compounds"?** Yes, but specifically: it compounds through *duplication*, not through tangled call graphs. Each new residential-vs-commercial or rental-vs-sale feature request has, so far, been solved by copying an existing page/component and adapting it, rather than parameterizing the original. That is a linear cost today (one more file to maintain) but a compounding cost going forward, because every twin pair is a place a fix applied to one side silently doesn't reach the other — which this report already found happening twice in Browse alone (favoriting-on-map, active-filters detection) without looking hard. The fix is not "add tests" (that's Track 7's job) — it's actively converging the twin pairs the same way `ListingsMapEnhanced` already converged the map (parallel props/data, one implementation) instead of letting the pattern keep replicating.
