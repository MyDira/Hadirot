# Track 1 — Database schema, RLS & migration integrity — Findings

## Summary

Overall the data layer is more disciplined than expected for an AI-built codebase: **all 48 public tables have RLS enabled**, privilege escalation on `profiles` is correctly blocked by trigger guards, every SECURITY DEFINER function has a pinned `search_path`, the admin analytics/inquiry RPCs enforce `require_admin()`/ownership internally, Stripe idempotency is partly guarded by unique indexes, price integrity is enforced by CHECK constraints, and the hot browse path has a well-matched composite index. **However there is one true P0**: the `listings` SELECT policy named "Hide listings from banned users" is a *permissive* policy whose predicate reduces to `true` for any non-banned authenticated user — because permissive policies are OR-combined, this silently grants every logged-in user read access to **all** listings including drafts, unapproved, deactivated rows and private owner fields (landlord financials, tenant notes, contact PII, full address). A second P0 class: three SMS/renewal tables (`sms_messages`, `listing_renewal_conversations`, `sms_admin_config`) carry `FOR ALL TO public USING true` policies (mislabeled "Service role full access"), so any anon or authenticated caller can read phone numbers + message bodies and insert/delete/overwrite them via the public REST API. The same permissive-policy misunderstanding makes the banned-user **INSERT** blocks non-functional (banned users can still post). Below that are constraint/idempotency gaps (no UNIQUE on `stripe_subscription_id`; `featured_purchases` missing the session-idempotency unique that `paid_listing_payments` has), FK ON DELETE choices that destroy payment records or block user deletion, a broken weekly-reports cron, and several low-sensitivity `USING true` public reads. Live verification against prod succeeded (48 tables, 140 policies, 91 SECURITY DEFINER funcs, 179 indexes, 54 FKs, 32 triggers, 48 CHECKs, pg_cron inspected); **both July-2 classifier-gated migrations are confirmed applied in prod — no local/live drift.** Severity counts: **2 P0, 2 P1, 8 P2, 4 P3.**

## Findings

### [P0] `listings` SELECT policy grants every authenticated user read access to ALL listings (drafts, unapproved, deactivated, private fields)
- **Where:** `listings` RLS policy `"Hide listings from banned users (except owners)"` (SELECT, PERMISSIVE, roles `{authenticated}`). Introduced by `supabase/migrations/20251030190000_final_security_fixes.sql` (and its `20251030182152_*` twin).
- **What:** The policy `USING ((user_id = auth.uid()) OR (NOT EXISTS (SELECT 1 FROM profiles WHERE profiles.id = auth.uid() AND profiles.is_banned = true)))` checks whether the **viewer** is banned, not the listing owner. For any non-banned authenticated viewer the second disjunct is `true`, so the whole `USING` is `true`. Postgres OR-combines permissive SELECT policies, so this blanket-`true` policy overrides the intended `"Anyone can read active rental listings"` restriction (`is_active AND approved AND type`). Net effect: any logged-in, non-banned user can `GET /rest/v1/listings?select=*` and receive every row.
- **Why it matters / failure scenario:** Registration is open (anyone can create a free account). A signed-in user can enumerate **all** listings regardless of `is_active`/`approved`/`payment_kind`, exposing: unapproved (pending-moderation) and rejected listings, deactivated listings, and private owner-only columns present on the row — `tenant_notes`, `rent_roll_data`, `rent_roll_total`, `contact_name`, `contact_phone`, `contact_phone_e164`, `full_address`. This is a confidentiality breach of landlord financial data and contact PII across the entire user base, trivially exploitable via the public REST endpoint. Gets worse with growth (1,384 listings today → all future listings).
- **Evidence:** Live `pg_policies` (all listings policies are `PERMISSIVE`, none `RESTRICTIVE`):
  - `SELECT | PERMISSIVE | Anyone can read active rental listings | {public}` → `USING ((is_active=true) AND (approved=true) AND (...type...))`
  - `SELECT | PERMISSIVE | Hide listings from banned users (except owners) | {authenticated}` → `USING ((user_id = auth.uid()) OR (NOT EXISTS (SELECT 1 FROM profiles WHERE id=auth.uid() AND is_banned=true)))`
  The migration author even documented the OR semantics ("Access is granted if ANY permissive policy allows it", lines 143-167) but did not realize this specific predicate reduces to `true`.
- **Fix prompt:** In a new migration, drop the permissive SELECT policy `"Hide listings from banned users (except owners)"` on `public.listings`. Ban-hiding of an *owner's* listings, if desired, must be either (a) enforced by deactivating listings when a user is banned (app/edge logic), or (b) implemented as a `RESTRICTIVE` policy that checks the **listing owner's** ban status, e.g. `CREATE POLICY "hide_banned_owner_listings" ON listings AS RESTRICTIVE FOR SELECT TO authenticated USING (user_id = auth.uid() OR NOT EXISTS (SELECT 1 FROM profiles p WHERE p.id = listings.user_id AND p.is_banned));`. Keep the `"Users can read own listings"`, `"Anyone can read active rental listings"`, and `"Admins can manage all listings"` policies. Verify by connecting as a plain authenticated (non-admin) test user and confirming `SELECT count(*) FROM listings` returns only active+approved+own rows, and that an inactive/unapproved listing belonging to another user returns zero rows. Do NOT reintroduce a permissive policy whose USING can evaluate to true independent of the row.

### [P0] SMS/renewal tables world-readable & world-writable via `FOR ALL TO public USING true` policies
- **Where:** RLS policies on `sms_messages` (`"Service role full access on sms_messages"`), `listing_renewal_conversations` (`"Service role full access"`), and `sms_admin_config` (`"Service role full access on sms_admin_config"`) — each `cmd=ALL`, `roles={public}`, `USING true`. Row counts: `sms_messages` 2,050; `listing_renewal_conversations` 671; `sms_admin_config` 1.
- **What:** These policies are named for `service_role` but were created `TO public` (or with no role, defaulting to public), so they apply to `anon` and `authenticated` too. Supabase grants `anon`/`authenticated` full DML on these tables by default (verified: both roles hold `SELECT,INSERT,UPDATE,DELETE,TRUNCATE`), and RLS is the only gate — a `USING true` ALL policy opens every operation. `service_role` already bypasses RLS, so restricting these to `service_role` costs nothing.
- **Why it matters / failure scenario:** Any anonymous caller with the public anon key can `GET /rest/v1/sms_messages?select=*` and read every SMS `phone_number` + `message_body` (real user PII / conversation content), or `GET /rest/v1/listing_renewal_conversations` for phone numbers, `reply_text`, and `message_sid`. They can also **write**: forge inbound SMS rows, flip `listing_renewal_conversations.state`/`action_taken`/`hadirot_conversion` (which the renewal flow acts on), delete conversation logs, or rewrite `sms_admin_config.admin_email` / disable `notify_on_errors` to blind the operator. The app itself never reads these tables from the client (grep of `src/` finds no `.from('sms_messages'|'sms_admin_config'|'listing_renewal_conversations')`), confirming the public grant is unintended.
- **Evidence:** Live `pg_policies`: `sms_messages [ALL] "..." roles={public} USING: true`; `listing_renewal_conversations [ALL] "Service role full access" roles={public} USING: true`; `sms_admin_config [ALL] "..." roles={public} USING: true`. Live `role_table_grants`: `sms_messages | anon | DELETE,INSERT,REFERENCES,SELECT,TRIGGER,TRUNCATE,UPDATE` (same for `authenticated` and for the other two tables). Columns confirm PII: `sms_messages(phone_number text, message_body text, message_sid text, ...)`, `listing_renewal_conversations(phone_number text, reply_text text, state text, action_taken text, ...)`.
- **Fix prompt:** In a new migration, for each of `sms_messages`, `listing_renewal_conversations`, `sms_admin_config`: `DROP POLICY` the mislabeled public policy and recreate it `TO service_role` (or simply drop it — service_role bypasses RLS, and add an admin-read policy `USING (is_admin_cached())` if the admin UI needs read access). Example: `DROP POLICY "Service role full access on sms_messages" ON sms_messages; CREATE POLICY "sms_messages_service_role" ON sms_messages FOR ALL TO service_role USING (true) WITH CHECK (true);`. Verify by querying each table with the anon key and confirming `permission denied` / zero rows, and confirm the Twilio edge functions (which use the service-role key) still read/write normally. Cross-check with Track 2 (security) which also covers the SMS webhook surface.

### [P1] Banned-user INSERT blocks are non-functional (banned users can still post rentals and commercial listings)
- **Where:** `listings` policy `"Prevent banned users from inserting listings"` (INSERT, PERMISSIVE) and `commercial_listings` policy `"Banned users cannot insert commercial listings"` (INSERT, PERMISSIVE).
- **What:** Same permissive-policy fallacy as the P0. Each is a permissive INSERT policy with `CHECK (NOT EXISTS (... profiles.is_banned ...))`, OR-combined with the normal `"Users can create rental listings"` / `"Authenticated users can insert own commercial listings"` policy whose `CHECK` is `(user_id = auth.uid())`. For a banned user inserting their own row, the second policy's CHECK is `true`, so the INSERT is allowed. A permissive policy can only *grant*; it cannot enforce a prohibition.
- **Why it matters / failure scenario:** A banned user is not actually prevented from posting new listings at the database level — the ban relies entirely on app-layer checks. If any code path (or a direct REST call with the user's own JWT) inserts a listing, the ban is bypassed. Given the site's abuse posture (SMS/email cost per listing event), a banned spammer can keep posting.
- **Evidence:** Live `pg_policies`: `listings [INSERT] "Prevent banned users from inserting listings" PERMISSIVE {authenticated} CHECK (NOT EXISTS(... is_banned ...))` alongside `listings [INSERT] "Users can create rental listings" PERMISSIVE {authenticated} CHECK (user_id = auth.uid())`. Postgres OR-combines permissive CHECKs.
- **Fix prompt:** In a new migration, convert the two ban-INSERT policies to `RESTRICTIVE` so they AND with the grant policy: `DROP POLICY "Prevent banned users from inserting listings" ON listings; CREATE POLICY "block_banned_inserts" ON listings AS RESTRICTIVE FOR INSERT TO authenticated WITH CHECK (NOT EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() AND is_banned = true));` and the analogous change on `commercial_listings`. Verify by attempting an insert as a banned test user (should fail) and as a normal user (should succeed).

### [P1] `featured_purchases` lacks the Stripe-session idempotency UNIQUE that `paid_listing_payments` has (webhook replay → double featured entitlement)
- **Where:** `featured_purchases` — index `idx_featured_purchases_stripe_session` on `stripe_checkout_session_id` is a **plain** (non-unique) index. Contrast `paid_listing_payments` which has `idx_paid_listing_payments_stripe_session_unique` UNIQUE partial index.
- **What:** There is a partial-unique guard preventing two *active* featured purchases per listing (`idx_featured_purchases_one_active_per_listing` on `listing_id WHERE status IN (pending,paid,active)`), but nothing enforces one row per Stripe checkout session. Stripe delivers webhooks at-least-once; without a unique key on `stripe_checkout_session_id`, a redelivered `checkout.session.completed` can insert a duplicate `featured_purchases` row for the same payment.
- **Why it matters / failure scenario:** At 1,000 paying users, occasional Stripe webhook redelivery is normal. A duplicate row can double-count a user against their featured cap, or extend/re-grant featured time twice for one payment (revenue/entitlement drift). The one-active-per-listing partial unique mitigates the worst case but not all statuses (e.g., two `paid`→`expired` rows, or accounting/reporting doublecounts).
- **Evidence:** Live `pg_indexes`: `featured_purchases: CREATE INDEX idx_featured_purchases_stripe_session ON ... (stripe_checkout_session_id)` (no UNIQUE) vs `paid_listing_payments: CREATE UNIQUE INDEX idx_paid_listing_payments_stripe_session_unique ON ... (stripe_checkout_session_id) WHERE stripe_checkout_session_id IS NOT NULL`.
- **Fix prompt:** Add a partial UNIQUE index mirroring the payments table: `CREATE UNIQUE INDEX CONCURRENTLY idx_featured_purchases_stripe_session_unique ON featured_purchases (stripe_checkout_session_id) WHERE stripe_checkout_session_id IS NOT NULL;` (first confirm no existing duplicates: `SELECT stripe_checkout_session_id, count(*) FROM featured_purchases WHERE stripe_checkout_session_id IS NOT NULL GROUP BY 1 HAVING count(*)>1;`). Ensure the `stripe-webhook` handler uses `ON CONFLICT (stripe_checkout_session_id) DO NOTHING/UPDATE`. Coordinate with Track 3 (billing).

### [P2] Maintenance/lifecycle SECURITY DEFINER functions are EXECUTE-able by `anon`/`authenticated`
- **Where:** `deactivate_old_listings()`, `delete_very_old_listings()`, `reconcile_individual_listing_anchors()`, `enforce_subscription_listing_cap()`, plus the `analytics`/`increment_*`/`touch_session` helpers — all `has_function_privilege('anon'/'authenticated','EXECUTE') = true`.
- **What:** `deactivate_old_listings()` and `delete_very_old_listings()` are SECURITY DEFINER cron functions (also scheduled via pg_cron) that inactivate/anonymize listings by fixed age criteria. They can be invoked by any anon/authenticated caller via `POST /rest/v1/rpc/deactivate_old_listings`. They don't take arbitrary targets (the WHERE clauses are fixed: `is_active=false AND deactivated_at < now()-30d`, etc.), so this is not arbitrary deletion, but a caller can force these batch jobs to run on demand.
- **Why it matters / failure scenario:** A malicious user can repeatedly trigger the cleanup/anonymization batch (extra write load, premature runs, and `delete_very_old_listings` performs storage deletes in a row loop). Lower severity because effects are bounded by the functions' own predicates, but these should be service-role/cron only. `touch_session`/`increment_listing_views` additionally let anon inflate view counts / forge session `user_id` (analytics poisoning).
- **Evidence:** Live `pg_proc` privilege check: `deactivate_old_listings() | anon=true | auth=true`, `delete_very_old_listings() | anon=true | auth=true`, `reconcile_individual_listing_anchors() | anon=true | auth=true`. By contrast `expire_featured_listings()`, `cleanup_analytics_events()`, `rollup_analytics_events()` are correctly `anon=false auth=false`. Migration `20260427000000_lock_down_anon_callable_functions.sql` locked some functions but missed these.
- **Fix prompt:** In a new migration, `REVOKE EXECUTE ON FUNCTION public.deactivate_old_listings(), public.delete_very_old_listings(), public.reconcile_individual_listing_anchors(), public.enforce_subscription_listing_cap() FROM anon, authenticated, public;` (pg_cron runs as a superuser/owner and is unaffected). Audit the full `anon=true` SECURITY DEFINER list and revoke EXECUTE from anything not intentionally called from the client; keep the analytics_* funcs (they gate on `require_admin()` internally) and the genuinely public helpers (`search_locations`, `get_sales_feature_enabled`, `increment_article_views`). Verify with `has_function_privilege('anon', 'public.delete_very_old_listings()', 'EXECUTE')` returning false.

### [P2] No UNIQUE on `stripe_subscription_id` — two rows can claim the same Stripe subscription
- **Where:** `listing_subscriptions.stripe_subscription_id` (only a plain partial index `idx_listing_subscriptions_stripe_sub`), and `concierge_subscriptions` (no stripe unique index at all).
- **What:** The master prompt asks explicitly "can two rows claim the same Stripe subscription?" — yes. `listing_subscriptions` has `idx_listing_subscriptions_one_active_per_user` (partial unique on `user_id WHERE status IN active/admin_active/past_due/trial`) which prevents two active subs per user, but nothing prevents two rows referencing the same `stripe_subscription_id` (e.g., a cancelled row + a new row, or a webhook double-insert of a fresh subscription).
- **Why it matters / failure scenario:** On webhook redelivery or a create-checkout double-click, duplicate subscription rows can be created that all point at one Stripe subscription, causing entitlement/reconciliation ambiguity (which row does `cascade-deactivate-subscription` update?).
- **Evidence:** Live `pg_indexes` for `listing_subscriptions`: `idx_listing_subscriptions_stripe_sub` is `CREATE INDEX ... WHERE stripe_subscription_id IS NOT NULL` (non-unique); no `UNIQUE` index references `stripe_subscription_id`. `concierge_subscriptions` indexes contain no `stripe_subscription_id` index.
- **Fix prompt:** After checking for existing dups, add `CREATE UNIQUE INDEX CONCURRENTLY idx_listing_subscriptions_stripe_sub_unique ON listing_subscriptions (stripe_subscription_id) WHERE stripe_subscription_id IS NOT NULL;` and the equivalent on `concierge_subscriptions` if it stores a Stripe subscription id. Ensure webhook upserts use `ON CONFLICT`. Coordinate with Track 3.

### [P2] FK ON DELETE choices: user deletion destroys payment records and is blocked by agency ownership
- **Where:** FKs `paid_listing_payments.user_id → profiles ON DELETE CASCADE`, `featured_purchases.user_id → auth.users ON DELETE CASCADE`, `listings.user_id → profiles ON DELETE CASCADE`; and `agencies.owner_profile_id → profiles ON DELETE RESTRICT`.
- **What:** Two problems. (1) Deleting a user (or their `auth.users` row) CASCADEs through `profiles → listings → paid_listing_payments / featured_purchases`, permanently deleting **financial/payment history**. `commercial_listings.user_id` uses `ON DELETE SET NULL` (preserves the listing) — inconsistent with the residential CASCADE. (2) `agencies.owner_profile_id` is `ON DELETE RESTRICT`, so deleting a profile that owns an agency **fails** unless the agency is first reassigned/deleted — the `delete-user` edge function will error for any agency owner.
- **Why it matters / failure scenario:** GDPR/"delete my account" or admin user-removal will (a) silently erase Stripe payment audit records needed for accounting/dispute handling, and (b) hard-fail for agency owners, leaving a partially-deleted user. At scale both surface as support incidents.
- **Evidence:** Live `pg_constraint`: `paid_listing_payments.paid_listing_payments_user_id_fkey -> CASCADE`, `featured_purchases.featured_purchases_user_id_fkey -> CASCADE (REFERENCES auth.users)`, `agencies.agencies_owner_profile_id_fkey -> RESTRICT`, vs `commercial_listings.commercial_listings_user_id_fkey -> SET NULL`.
- **Fix prompt:** Decide the account-deletion policy with the owner, then in a migration: change `paid_listing_payments.user_id` and `featured_purchases.user_id` FKs to `ON DELETE SET NULL` (preserve financial rows; make the columns nullable) or move payment history to an append-only ledger before delete; and either change `agencies.owner_profile_id` to `ON DELETE SET NULL` or have `delete-user` reassign/deactivate owned agencies first. Verify by simulating a delete of a test user who owns an agency and has payments in a transaction you roll back. Cross-check Track 7 (`delete-user` PII completeness) and Track 3.

### [P2] `send-weekly-performance-reports` cron job is broken (literal placeholders) — reports silently never send
- **Where:** pg_cron job `send-weekly-performance-reports` (schedule `0 19 * * 4`).
- **What:** The job command posts to `https://[PROJECT-REF].supabase.co/functions/v1/send-weekly-performance-reports` with header `Authorization: Bearer [SERVICE-ROLE-KEY]` — the placeholders were never substituted. Every Thursday-7pm invocation fails (bad host / bad auth), so weekly performance reports have never been sent.
- **Why it matters / failure scenario:** A feature the owner believes is running produces nothing, with no error surfaced. Classic "owner won't know until someone asks where the report is."
- **Evidence:** Live `cron.job`: `send-weekly-performance-reports | 0 19 * * 4 | SELECT net.http_post(url:='https://[PROJECT-REF].supabase.co/functions/v1/send-weekly-performance-reports', headers:='{"Content-Type": "application/json", "Authorization": "Bearer [SERVICE-ROLE-KEY]"}'::jsonb, ...)`.
- **Fix prompt:** Re-create the cron job with the real project ref and a service-role bearer (or, better, the `current_setting('app.supabase_url')` / `current_setting('app.supabase_service_role_key')` pattern already used by the `send-paid-listing-reminders` job). Verify with a manual `SELECT net.http_post(...)` and confirm the edge function receives it. Coordinate with Track 7 (cron/ops).

### [P2] Service-role JWTs hardcoded inside pg_cron job commands
- **Where:** pg_cron jobs `cleanup-expired-renewals`, `send-renewal-reminders` embed a full `role:"service_role"` JWT in the `Authorization` header; `cleanup-expired-drafts-daily` embeds an `anon` JWT.
- **What:** Long-lived service-role tokens are stored in plaintext in the `cron.job` catalog. Not exposed via PostgREST (the `cron` schema isn't API-reachable), but any DB-level read (or a leaked catalog dump / migration file containing the same literal) exposes a service-role key that bypasses all RLS.
- **Why it matters / failure scenario:** If these same literals also live in committed migration/SQL files (see Track 2's secret scan), the service-role key is effectively in git history. Rotating it later requires editing every cron job.
- **Evidence:** Live `cron.job` command for `send-renewal-reminders` contains `Authorization: Bearer eyJhbGciOiJIUzI1NiI...role":"service_role"...`; `cleanup-expired-renewals` likewise; `cleanup-expired-drafts-daily` uses an `anon` JWT literal.
- **Fix prompt:** Move the tokens out of the job body: store the service-role key via `ALTER DATABASE ... SET app.supabase_service_role_key = ...` (as the `send-paid-listing-reminders` job already does) and reference `current_setting('app.supabase_service_role_key')`. Then rotate the exposed service-role key in the Supabase dashboard. Primary owner: Track 2 (secrets) — noted here because the exposure lives in a DB object.

### [P2] `scrape_runs` intended admin-only, but readable by any authenticated user
- **Where:** `scrape_runs` policy `"Admins can read scrape runs"` (SELECT, roles `{authenticated}`, `USING true`). 17,079 rows.
- **What:** The policy is named admin-only but its `USING` is literally `true`, so every authenticated user can read all scrape-run metadata (`source`, `pdf_filename`, `errors` jsonb, counts). Anon cannot (no anon policy), so this is authenticated-wide, not public.
- **Why it matters / failure scenario:** Intent violated; the `errors` jsonb and source filenames leak operational detail to any logged-in user. Low sensitivity but clearly not what "Admins can read" implies, and the table grows unbounded (17k rows already).
- **Evidence:** Live `pg_policies`: `scrape_runs [SELECT] "Admins can read scrape runs" roles={authenticated} USING: true`. `src/services/pipeline.ts:35` reads `.from('scrape_runs')` from an admin UI, but the RLS doesn't restrict to admins.
- **Fix prompt:** Replace the `USING true` with `USING (is_admin_cached())` (matching the pattern used by `scraped_listings`): `DROP POLICY "Admins can read scrape runs" ON scrape_runs; CREATE POLICY "Admins can read scrape runs" ON scrape_runs FOR SELECT TO authenticated USING (is_admin_cached());`. Verify a non-admin authenticated user gets zero rows and the admin pipeline UI still works.

### [P2] Low-sensitivity public reads via `USING true` (`analytics_sessions`, `modal_user_interactions`, `admin_settings`, daily aggregates)
- **Where:** `analytics_sessions` (`"as_read_all"` SELECT `{public}` USING true, 47,240 rows), `modal_user_interactions` (`"Anyone can read modal interactions"` SELECT `{public}` USING true, 24,401 rows), `admin_settings` (`"Anyone can read admin settings"` SELECT `{public}` USING true), `daily_analytics`/`daily_top_filters`/`daily_top_listings` (public USING true).
- **What:** These grant anonymous read. `analytics_sessions` exposes `user_id ↔ session_id ↔ anon_id` linkage + timing (behavioral privacy); `modal_user_interactions` exposes `user_id` + `page_path` + interaction history; `admin_settings` exposes business config (`charge_agents`, `monetization_enabled`, caps). The daily_* tables are pre-aggregated and low-risk. `admin_settings` is partly needed by the frontend (e.g., `sales_feature_enabled`) so full-row public read is over-broad rather than wholly wrong.
- **Why it matters / failure scenario:** Not a credential/PII-content leak, but an anonymous user can scrape user-behavior linkage (`analytics_sessions`/`modal_user_interactions`) and read internal toggles (`charge_agents`). Row counts grow unbounded.
- **Evidence:** Live `pg_policies` for each table (USING true, roles `{public}`); columns: `analytics_sessions(session_id, anon_id, user_id, ...)`, `modal_user_interactions(modal_id, user_id, page_path, ...)`, `admin_settings(..., charge_agents, monetization_enabled, ...)`.
- **Fix prompt:** For `analytics_sessions` and `modal_user_interactions`, drop the public SELECT policy and keep only the admin SELECT policy (writes are done by SECURITY DEFINER funcs / service role, so no client read is needed — confirm no `.from('analytics_sessions')` in `src/`; grep shows none). For `admin_settings`, narrow the public policy to expose only the flags the frontend needs (a SECURITY DEFINER getter like the existing `get_sales_feature_enabled()` is the cleaner pattern) or accept as-is if the owner considers these non-secret. Verify anon read returns zero for the two analytics tables.

### [P3] Missing composite index for non-default browse sorts (price / bedrooms)
- **Where:** `listings` browse query in `src/services/listings.ts` (`getListings`, lines ~170-258). Default sort `created_at DESC` is covered by `listings_browse_idx (listing_type, is_active, approved, created_at DESC)`; price/bedrooms/bathrooms sorts are not.
- **What:** When the user picks `price_asc/desc` or `bedrooms_*`, the query filters via `listings_browse_idx` but then performs an in-memory sort of the active+approved set. Range/`IN` filters on `price`, `bedrooms`, `neighborhood` are residual (no supporting index on `listings.neighborhood`).
- **Why it matters / failure scenario:** Fine today (1,384 rows). At ~100k listings with ~20k active, each price-sorted page load sorts ~20k rows and neighborhood-filtered browses do a partial scan — noticeable latency, not breakage.
- **Evidence:** Live `pg_indexes` for `listings`: composite exists only for `created_at DESC`; no `(listing_type, is_active, approved, price)` and no `listings(neighborhood)`. Query uses `.order('price', ...)`, `.in('neighborhood', ...)`, `.in('bedrooms', ...)`.
- **Fix prompt:** When list volume approaches ~50k, add covering composites for the offered sorts, e.g. `CREATE INDEX CONCURRENTLY idx_listings_browse_price ON listings (listing_type, is_active, approved, price) WHERE is_active AND approved;` and consider `listings (neighborhood)` for the neighborhood filter. Validate with `EXPLAIN ANALYZE` on a price-sorted browse at scale. Not urgent at current volume.

### [P3] Minor unindexed FK columns (notably `favorites.listing_id` path)
- **Where:** `favorites.listing_id` is only covered by the composite `(user_id, listing_id)` (not leading), plus small admin tables (`digest_*`, `concierge_*.user_id`, `scraped_listings.*`).
- **What:** `favorites.listing_id` has no leading index, so `ON DELETE CASCADE` when a listing is removed, and any "who favorited this listing" query, scans `favorites`. (Note: migration `20251030190000` claims to add `favorites_listing_id_idx` with `IF NOT EXISTS`, but live `pg_indexes` shows it is not present — either dropped later or never applied.)
- **Why it matters / failure scenario:** Negligible now (197 rows); grows with engagement. At 50k users the CASCADE/lookup does a growing seq scan.
- **Evidence:** FK-vs-index cross-check (script): `favorites.listing_id (rows=197)` flagged as having no leading index; live `pg_indexes` for `favorites` shows only `favorites_pkey` and `favorites_user_id_listing_id_key`.
- **Fix prompt:** `CREATE INDEX CONCURRENTLY favorites_listing_id_idx ON favorites (listing_id);`. Optionally add leading indexes on the other flagged FK columns if those tables grow. Verify with `EXPLAIN` on a listing delete.

### [P3] `geocode_cache` has RLS enabled with zero policies (service-role-only) — verify intentional
- **Where:** `geocode_cache` (RLS enabled, 0 policies, 600 rows).
- **What:** With RLS on and no policy, all `anon`/`authenticated` access is denied (silently). Only `service_role` (bypass) can read/write. This is correct IF only edge functions use it. Grep of `src/` finds no client `.from('geocode_cache')`, so it appears intentional — flagged only so "RLS-on-no-policy" isn't mistaken for an oversight.
- **Why it matters / failure scenario:** No risk today; would only bite if a future client feature tries to read the cache directly and silently gets nothing.
- **Evidence:** Live table list: `geocode_cache | rls=true | pol=0 | rows=600`. No client references.
- **Fix prompt:** No action required; if a client read is ever needed add an explicit `SELECT` policy. Otherwise leave as-is (documented intentional).

### [P3] `touch_session` / `increment_listing_views` allow anonymous analytics/view-count manipulation
- **Where:** SECURITY DEFINER funcs `touch_session(p_session, p_anon, p_user, p_ts)` and `increment_listing_views(listing_id)`, both `anon=true`.
- **What:** `touch_session` inserts/updates `analytics_sessions` with a caller-supplied `user_id`, so an anon caller can forge a session attributed to any user. `increment_listing_views` bumps `listings.views` and can be called in a loop to inflate view counts.
- **Why it matters / failure scenario:** Analytics/vanity-metric poisoning; no security/financial impact. Bounded but trivially abusable.
- **Evidence:** Live `pg_proc` privilege: both `anon=true auth=true`; `touch_session` body writes `analytics_sessions.user_id := p_user` with no auth check.
- **Fix prompt:** Low priority. If analytics integrity matters, gate `increment_listing_views` behind a rate limit at the `track`/edge layer and have `touch_session` derive `user_id` from `auth.uid()` instead of a parameter. Coordinate with Track 4/6 (analytics ingest).

## Non-findings (checked and confirmed healthy)

- **RLS coverage complete:** all 48 public tables have `relrowsecurity = true`. No user-data table is missing RLS. Only `geocode_cache` has zero policies, and that is intentional service-role isolation (P3 note above).
- **`profiles` privilege escalation is blocked.** Despite `authenticated` holding UPDATE grant and a self-update policy (`auth.uid() = id`), the BEFORE-UPDATE trigger `prevent_privileged_profile_updates` and BEFORE-INSERT `prevent_privileged_profile_inserts` (both SECURITY DEFINER, `search_path` pinned) raise/zero out any non-admin non-service-role attempt to change `is_admin`, `is_banned`, `can_feature_listings`, `max_featured_listings_per_user`, `can_manage_agency`, `can_post_sales`. A user cannot self-elevate to admin or self-unban.
- **Admin checks do not trust client-supplied values.** `is_admin_cached()` reads the server-controlled JWT `app_metadata.is_admin` claim (synced from `profiles.is_admin` by the `profiles_sync_is_admin_to_auth` trigger) with a `profiles` lookup fallback — the client cannot forge `app_metadata`. Tables using the inline `EXISTS(profiles WHERE id=auth.uid() AND is_admin)` variant (`listing_subscriptions`, `paid_listing_payments`, `paid_listing_refunds`) are equally safe.
- **Admin-only SECURITY DEFINER data functions gate internally.** Verified bodies: `analytics_contact_submissions` (raises unless `is_admin`), `analytics_summary`/`analytics_inquiry_timing_phones` (call `require_admin()`), `get_listing_inquiries` (checks listing ownership before returning contact PII), `get_owner_listing_inquiry_counts` (scoped to `auth.uid()`). So even though many analytics RPCs are anon-EXECUTE-able, they do not leak data.
- **All 91 SECURITY DEFINER functions have a pinned `search_path`** (`public`, `public,pg_catalog`, or `public,pg_temp`) — no mutable-search-path hijack surface.
- **Listing draft/inactive visibility is otherwise correct** — separate from the P0 policy bug, the design intent (owner sees own; public sees active+approved; admin sees all) is present via distinct policies; the only defect is the over-broad "Hide banned" policy.
